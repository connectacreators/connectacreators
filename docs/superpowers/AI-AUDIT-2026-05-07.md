# AI Chat System Audit — 2026-05-07

Read-only audit of `supabase/functions/companion-chat/*` and the three frontend chat surfaces (`CompanionDrawer`, `CommandCenter`, `CanvasAIPanel`). Recent fixes are confirmed in place; this report focuses on what remains.

---

## Executive summary — the five things to fix first

1. **Tenant isolation hole on URL-locked client** (BLOCKER). `companion-chat/index.ts:478-485` looks up `urlClientId` with `.eq("id", urlClientId)` only — no `.eq("user_id", user.id)`. A logged-in user who navigates to `/clients/<other-tenants-uuid>/...` becomes that client in the AI's eyes. Every tool call after that can read/write the wrong tenant.
2. **Thread hijack via `incomingThreadId`** (HIGH). `index.ts:504` accepts `thread_id` from the request body and writes to it via `dualWriteCompanionTurn` without verifying the thread is owned by `user.id`. `appendMessage` (`_shared/assistant/threads.ts:110`) doesn't validate either. A malicious client can inject messages into anyone's thread.
3. **Cross-thread history bleed** (HIGH). `index.ts:563` loads `priorMessages` from `companion_messages` filtered only by `client_id`. Every new chat the user opens sees the same flat 40-message history for that client, even though the UI presents them as distinct threads. The model gets confused context, the user gets ghost answers from a previous chat.
4. **Canvas AI panel is on a separate edge function** (MEDIUM but surprising). `CanvasAIPanel.tsx` calls `ai-assistant` and `ai-build-script` (lines 698, 822, 1397, 1461), not `companion-chat`. None of the recent companion-chat fixes (5-round loop, removed "On it." fallback, eq user_id, isOnAiSurface) apply on the canvas. If the user reports "it works in the drawer but not in the canvas", this is why.
5. **`urlClientId` only constrains the active-client load, not tool calls** (MEDIUM). The system prompt rule 18b warns Claude, but tools that take `client_name` (every Wave 2/3/4 tool) still match by `ilike("name", "%X%")` within the user's tenant. On the agency `/ai` page or on a client URL, Claude can silently operate on a different client of the same user if the name partially matches.

## BLOCKER

### B1. URL-clientId lookup not user-scoped — cross-tenant takeover
- File: `supabase/functions/companion-chat/index.ts:478-485`
- Why it leaks: An authenticated user crafting (or being phished into) a URL like `/clients/<victim-uuid>/scripts` will have the edge function load that client (`.eq("id", urlClientId)` with no `user_id` filter). Once `client` is set to the victim's record, every downstream tool that uses `client.id` (memory, canvas, fill_onboarding, save_memory, dualWriteCompanionTurn, etc.) writes to the victim's account.
- Fix: add `.eq("user_id", user.id)` to the urlClient query. If it returns null, fall back to ownClient (current behavior) and log the attempted access.

## HIGH

### H1. Thread injection via untrusted `thread_id`
- Files: `supabase/functions/companion-chat/index.ts:504`, `_shared/assistant/threads.ts:110-127`
- Why it leaks: `incomingThreadId` is taken straight from the request body and used as-is to dual-write `assistant_messages`. `appendMessage` does no ownership check. Any authenticated user can POST `{ message, thread_id: "<somebody-elses-thread-uuid>" }` and pollute that thread's transcript.
- Fix: before trusting `incomingThreadId`, run `select id from assistant_threads where id = ? and user_id = ?`. If null, ignore it and create a fresh thread.

### H2. Cross-thread message bleed in `priorMessages`
- File: `supabase/functions/companion-chat/index.ts:560-572`
- Why it leaks: history is loaded from `companion_messages` keyed by `client_id` only. The frontend models this as multiple threads (`assistant_threads`), but the backend prompt smashes the last 40 messages from ALL threads on that client into one history. Users see "the AI remembers things from a different chat" — and worse, an old half-completed tool intent can nudge the model into re-acting.
- Fix: load history from `assistant_messages` filtered by `thread_id`. Drop `companion_messages` as the prompt source (keep it for the legacy companion bubble if still needed).

### H3. Tool descriptions claim writes that the LLM may execute under the wrong client
- File: `supabase/functions/companion-chat/index.ts:637` + `tools/leads.ts`, `tools/scripts.ts`, `tools/editing.ts`, `tools/research.ts`
- Why it leaks: when `urlClientId` is set, the system prompt warns Claude "Every tool call that takes client_name MUST use the locked name". But `resolveClient` (`tools/types.ts:32-46`) uses `ilike("name", "%X%")` — a fuzzy match. If Claude misreads the client name (e.g., user mentions "Gym Bros" but locked client is "Gymboree"), the tool happily resolves to whichever match comes first in tenant order. There is no enforcement. The user said the locked-URL surface should be sticky.
- Fix: thread `urlClientId` into `ToolContext` and have `resolveClient` short-circuit: if `urlClientId` set, return that client unconditionally and IGNORE the `client_name` argument. Log a warning when names disagree.

### H4. `MAX_ROUNDS=5` truncation produces silent stalls
- File: `supabase/functions/companion-chat/index.ts:710-1388`
- Why it leaks: when the loop exits because `round === MAX_ROUNDS - 1` and the last response was still `stop_reason: "tool_use"`, the user gets whatever `reply` was last set (often the round-1 text or `turn1Reply`). The model never got to write a final reply. The user sees a cropped response without knowing the AI was mid-action.
- Fix: when the final round still has `tool_use`, append the tool results and call Claude one more time with `tool_choice: {type: "none"}` to force a final text reply. Or surface a deterministic "I ran several actions; here's what's left to confirm…" message that lists the actions taken.

### H5. Anthropic API errors mid-loop log but don't propagate
- File: `supabase/functions/companion-chat/index.ts:730-734`
- Why it leaks: On `!apiRes.ok` mid-loop, the code sets `reply = "Anthropic returned an error..."` only when `!reply`. If Claude already wrote a partial reply in round 0 then round 2 fails with 5xx, the user sees the partial reply with no indication something failed. The dual-write also persists this misleading reply to the thread.
- Fix: always overwrite reply (or append a clear marker) when the loop exits via API failure. Consider HTTP 502 instead of dual-writing a half-finished assistant message.

## MEDIUM

### M1. `companion_messages` history loses tool_use/tool_result pairs
- File: `supabase/functions/companion-chat/index.ts:563`, `685-689`, `1395-1399`
- Why it's brittle: only plain text `reply` is saved. The next request rebuilds `messages = [...priorMessages, {user msg}]` and calls Anthropic — but if the FIRST message in priorMessages happens to be an assistant message (very common when you've had a chat that ended on the assistant), the API rejects with 400 ("first message must be user"). There is no leading-assistant-strip on this path. The canvas panel does strip (line 1342) but companion-chat does not.
- Fix: same as the canvas panel — find the first user message and slice from there before sending to Anthropic.

### M2. `read_canvas` can return very large payloads
- File: `supabase/functions/companion-chat/canvasReader.ts:39-61`
- Why it's brittle: caps each note at 800-1000 chars but iterates up to 12 text + 8 research + 6 media = 26 entries. Worst case ~22k chars in a single tool_result. Combined with prior messages (also unbounded), one over-loaded canvas + a long history can blow past the input window before round 2 even fires. There is no token accounting.
- Fix: cap total output at ~6k chars and emit "(N more notes truncated)" at the end. Check feed length before adding each.

### M3. Canvas AI panel uses a separate code path
- Files: `src/components/canvas/CanvasAIPanel.tsx:698, 822, 903, 1397, 1461`
- Why it's surprising: `ai-assistant` and `ai-build-script` are different edge functions with their own system prompts and tool catalogs. None of the companion-chat fixes carry over. If the user thinks "the AI" is one thing, they will be confused when one surface knows about Wave 2 tools (leads, finances, weekly priorities) and the other doesn't.
- Fix or accept: either route the canvas chat through `companion-chat` too, or call out in onboarding that "the canvas AI is a focused script-building variant; for everything else use the drawer or /ai".

### M4. Build-mode trigger is narrow
- File: `supabase/functions/companion-chat/build-mode.ts:142`
- Regex: `\b(let'?s\s+)?(build|write|create|make)\s+(me\s+)?a?\s*script\b`
- Misses: "draft a script", "give me a script", "I need a script", "script idea about X", "do me a script". Single-URL drops without an explicit verb work via `isFrameworkFirstTrigger`, but the worded variants do not.
- Fix: broaden to `(build|write|create|make|draft|need|give\s+me|generate)` and add `script(s)?` for plurals. Test "draft a script", "give me 3 scripts".

### M5. Build session can survive a totally unrelated message
- File: `supabase/functions/companion-chat/build-mode.ts:649-672`
- Why: routing checks "is there an active build session for this thread?" If yes, ALL future messages on that thread go to build mode regardless of content. A user who started a build, then asked "what's my pipeline?" gets routed into build-mode where there's no `get_pipeline_summary` tool and Claude has to improvise.
- Fix: when an active session exists but the new message has no URL and doesn't match build vocabulary, suspend the session for that turn (don't cancel) and route to companion-chat. Resume on next build-flavored message.

### M6. `update_client_strategy` and finance categories pass straight to DB
- Files: `supabase/functions/companion-chat/index.ts:1267-1283`, `tools/finances.ts:118-130`
- Why it's a footgun: no validation of percent sums (mix_reach + mix_trust + mix_convert can total 173%), no clamping of negatives, no check that revenue values are non-negative. `update_client_strategy` upserts whatever Claude provides. Claude has a track record of inventing numbers when asked to "balance the mix".
- Fix: clamp `mix_*` to 0-100, ensure they sum to ≤100 (warn if not), reject negative budgets/goals.

### M7. `dualWriteCompanionTurn` writes the FINAL reply only — progress lost
- File: `supabase/functions/companion-chat/index.ts:406-429`
- Why it's brittle: rounds 0-3 of multi-turn might produce intermediate text (e.g., "Looking up your pipeline…" then "Found 12 leads, here's the breakdown"). Only the last `reply` is dual-written. The drawer's Realtime subscription on `assistant_messages` therefore never sees mid-stream narration. Users perceive it as "AI went silent then suddenly answered".
- Fix: stream is a bigger refactor; the cheap win is to insert each round's text into `assistant_messages` as a "progress" content type so the drawer Realtime renders it.

### M8. `priorMessages` uses raw Supabase content (could be string OR object)
- File: `supabase/functions/companion-chat/index.ts:569-572`
- Why it's brittle: `companion_messages.content` is a TEXT column historically containing raw strings, BUT the dual-write inserts an object into `assistant_messages.content` (`{type: "text", text: ...}`). If anything ever cross-pollinates the two tables, Anthropic gets a content shape it can't parse.
- Fix: tighten the map at line 571 to coerce `content` to a string before sending to Anthropic.

## LOW

### L1. Action types referenced in tools but not handled in CommandCenter/Drawer
- File: `src/pages/CommandCenter.tsx:277-307`, `src/components/CompanionDrawer.tsx:238-269`
- Both surfaces handle `navigate`, `fill_onboarding`, `open_client`, `refresh_data`, `show_notification`. The backend never emits anything else, so this is fine — but if a future tool emits a new action type it will be silently dropped. Add a `console.warn("[ai] unhandled action type:", action.type)` in the else branch on both surfaces so future regressions are visible.

### L2. `get_weekly_priorities` does N+1 queries
- File: `supabase/functions/companion-chat/tools/intelligence.ts:95-130`
- For 10 clients it issues 50 sequential-ish promises. Acceptable today but will slow as client count grows. Not a leak, just a footgun for the Sonnet 5xx timeout window.

### L3. `find_viral_videos` searches caption substring with no full-text index
- File: `supabase/functions/companion-chat/index.ts:840`
- `ilike("caption", "%topic%")` on a likely-large `viral_videos` table. If the table grows past 100k rows, this becomes the slowest tool in the chain. Add a tsvector + GIN index, or push viral search to a dedicated edge function.

### L4. `respond_to_user` content lost when block.input.message is empty
- File: `supabase/functions/companion-chat/index.ts:751-756`
- `reply = block.input.message || "";` — if Claude calls `respond_to_user` with an empty string, the existing reply (if any) is overwritten with "". Then `if (!reply)` later sets it to a generic fallback. Cheap fix: only assign when the input is a non-empty string.

### L5. `MAX_ROUNDS = 6` for build-mode but 5 for companion-chat
- Files: `build-mode.ts:32`, `index.ts:710`
- Inconsistent budgets. Not wrong, but easy to forget when debugging. Document why one is 6 and the other 5, or unify.

### L6. `console.error/warn` go to Supabase function logs only
- Across companion-chat. The dev sees them in the Supabase dashboard but the user has no UI for "why did this fail?" The fallback string ("Let me try again — could you rephrase that?") is opaque.
- Fix (longer-term): when the loop fails for a known reason (Anthropic 5xx, missing client, RLS error), surface a one-line diagnostic prefix to the user: "Anthropic timed out — try again." vs the generic fallback.

## FYI / observations

- The 5-round multi-turn loop, removed hardcoded fallback, unhandled-tool fallback, isOnAiSurface gating, and most user-scoped client lookups all read clean. Good work.
- System prompt rules 1-19 are coherent; no contradictions found. Rule 18 + 18b are the right call for the script/canvas workflow.
- Tool catalog has no name collisions; every TOOLS entry has a matching handler (verified by inspection — `respond_to_user`, `create_script`, `get_hooks`, `list_all_clients`, `find_viral_videos`, `list_client_scripts`, `schedule_content`, `submit_to_editing_queue`, `get_editing_queue`, `get_content_calendar`, `create_canvas_note`, `read_canvas`, `add_video_to_canvas`, `add_research_note_to_canvas`, `add_idea_nodes_to_canvas`, `add_script_draft_to_canvas`, `save_script_from_canvas`, `get_client_strategy`, `update_client_strategy`, `get_client_info`, `navigate_to_page`, `save_memory`, `fill_onboarding_fields`, `open_client` are all handled in index.ts; modules cover the remaining names).
- The `refresh_data` listeners exist on ContentCalendar, EditingQueue, Scripts, ContractsPage, Finances, LeadTracker — solid coverage. Worth verifying they each match the scope strings the backend emits (`scripts`, `editing_queue`, `calendar`, `leads`, `finances`, `contracts`).
- `dualWriteCompanionTurn` is non-blocking on failure, which is the right call. Just needs the user_id check on the threadId.

## Suggested capability additions (only if they materially expand usefulness)

These are tools a power-user agency owner would expect that don't exist yet:

- **`get_post_performance(client_name, days_back)`** — pull engagement stats from `viral_videos` rows where the channel matches the client. Closes the "which of my posts is underperforming?" loop.
- **`compare_clients(client_a, client_b, metric)`** — side-by-side strategy + actuals. The data is all there in `client_strategies` and the count queries.
- **`get_recent_activity(days_back)`** — flat timeline of "what changed in the last N days" across scripts/leads/edits/posts. Most useful single tool for "catch me up".
- **`bulk_reschedule_posts(client_name, from_date, to_date, shift_days)`** — current `reschedule_post` is one at a time. Power users with 10+ scheduled posts will hate it.
- **`search_messages(query)`** — full-text search across the user's `assistant_messages`. "Find the chat where I saved that hook idea last week."

The first three are pure-read and small. Worth doing first.
