# AI Chat System Audit — 2026-05-08

Read-only review of the companion-chat surface after today's ~25-change drop.
Covers `supabase/functions/companion-chat/**`, `CompanionDrawer`,
`CommandCenter`, `useActiveChat`, and the assistant primitives. No code
modified.

---

## Top 5 things to fix first

1. **Inline script preview card shows "Untitled Script" and a blank body on
   `/ai`.** `parseScriptDraft` in CommandCenter.tsx:444-491 builds a
   `script_data` object with `{hook, body, cta, idea_title}`, but
   `InlineScriptPreview` (CanvasAIPanel.shared.tsx:323-393) reads
   `script.idea_ganadora` and iterates `script.lines`. Wrong shape → broken
   card. The "title threaded through" claim is half-shipped.
2. **Most agency-cutting tools ignore admin/subscriber access entirely.**
   `get_all_clients_status`, `get_weekly_priorities`, `get_recent_activity`,
   `get_morning_brief`, `get_overdue_items`, `get_revenue_vs_goal`,
   `propose_plan`'s name lookup, and `open_client` all hard-filter on
   `clients.user_id = userId`. An admin viewing the agency from a non-owner
   account sees zero data; a subscriber won't see the clients they're
   assigned to. Inconsistent with the well-done access work in
   `resolveClient`/`get_open_alerts`.
3. **`save_script_from_canvas` (index.ts:1348-1384) bypasses the
   duplicate-check + rollback that `build-tool-handlers.handleSaveScript`
   added.** Same write path, none of the safety. It can still create
   line-less scripts (and used to be the source of the "ghost script" bug).
4. **`generate_caption` writes Markdown asterisks/headers when the system
   prompt rule 1 forbids them everywhere else.** Haiku is unconstrained in
   `editing.ts:185-243` — the prompt says "no hashtags" but never says
   "plain text only", so captions land with `**bold**` and emojis the
   creator doesn't want.
5. **Plan-proposal flow has no expiry / orphan cleanup.** `pending_plans`
   has no TTL; nothing dismisses old proposals; `latestPlan` in
   CommandCenter is not hydrated from the DB on reload, so an approved-mid
   refresh leaves a never-clearable row. Also: the model can call
   `propose_plan` then immediately execute steps in the same turn — there
   is no server enforcement that destructive tools refuse without an
   approved plan.

## Findings by severity

### BLOCKER

**B1. Inline script preview card is broken on /ai.**
`src/pages/CommandCenter.tsx:464-475` synthesizes `script_data: {hook, body,
cta, idea_title}`. `src/components/canvas/CanvasAIPanel.shared.tsx:323-393`
reads `script.idea_ganadora` (title) and `script.lines[]` (rendered body).
The card renders "Untitled Script" with no preview. Save button still
fires `handleApproveScript` ("Approve and save…"), so save works — but the
visual is broken. Fix: either rename the synthesized fields to match
`ScriptResult` (idea_ganadora + lines[{section, text, line_type}]) or
extend InlineScriptPreview to accept the simpler `{hook, body, cta}`
shape.

**B2. `save_script_from_canvas` bypasses every safeguard added today.**
`index.ts:1348-1384` does the script_lines insert without checking the
return error and without rollback if it fails. No duplicate detection. No
line-content validation. Same bug class that produced the 3 ghost
scripts. The non-canvas `create_script` (index.ts:912-956) has the same
gap. Promote `handleSaveScript`'s validate-then-rollback to a shared
helper; both inline branches should use it.

### HIGH

**H1. Agency/subscriber scoping ignored in seven cross-cutting tools.**
- `tools/intelligence.ts:67-71, 115` — `get_all_clients_status` and
  `get_weekly_priorities` `.eq("user_id", userId)`. An agency owner's
  staff (subscriber) sees nothing.
- `tools/analytics.ts:229, 350, 380` — `get_recent_activity`,
  `get_morning_brief`, `get_overdue_items` all do the same.
- `tools/finances.ts:165` — `get_revenue_vs_goal` only iterates
  user_id-owned clients (financial transactions are `user_id`-scoped, but
  the goal aggregation should respect the access set or admins won't see
  agency-wide totals).
- `tools/plans.ts:80-86` — `propose_plan`'s client name lookup is global
  (no scoping at all — worse than the others). Could resolve to a client
  in another tenant.
- `index.ts:1544-1559` — `open_client` only `.eq("user_id", user.id)`.
  Admin can't navigate to a client they don't own from the AI.

Why it matters: the user is the agency owner. They added the agency-aware
access model precisely so this stuff works. Half the new tools didn't get
the memo. Fix: pull the access filter from `ctx.isAdmin` /
`ctx.accessibleClientIds` (already on every ToolContext), same pattern as
`get_open_alerts:194-207`.

**H2. `propose_plan` is advisory only — no server enforcement.**
The model can call `propose_plan` AND `delete_script` in the same round;
nothing forces it to wait. Today this is a system-prompt rule (18c) the
model may follow or skip. Real fix: a `requires_plan_approval = true` flag
on the destructive tools, checked against an `approved` row in
`pending_plans` for this thread within the last N minutes. Without that,
"PREVIEW BIG ACTIONS" is documentation, not a guardrail.

**H3. `latestPlan` state in CommandCenter is ephemeral and not
DB-backed.** `CommandCenter.tsx:151-153` → if the user reloads the page
between propose and approve, the card disappears. The approve/reject
buttons send synthetic chat messages, so the action still works — but the
user has no way to find the proposal again. Add: load any pending plan
for the active thread on mount.

**H4. `submit_to_editing_queue` (index.ts:1096-1119) and
`submit_to_editing_after_save` (build-mode) collide.** Both insert a
`video_edits` row; the model has to pick one. Description differences
are subtle. Outside of a build session the model defaults to the inline
one and bypasses build-mode's nicer "after_save" semantics. Either
deprecate the inline tool, OR make it forward to the same handler so
behavior is identical.

**H5. `findEditItem` matches by ilike substring with no
client_id-aware safety.** `tools/editing.ts:109-119` is fine — but if
the AI calls `mark_post_published` with `item_title="Reel"` (very generic
on chatty users), it'll grab the first matching reel and mark it
published. Combined with H2 (no plan enforcement), an over-eager `auto`
mode session can publish the wrong post. Fix: refuse to act if titlepartial
length < 5 chars OR > 1 candidate matches.

**H6. `dismiss_alert` doesn't refresh the system-prompt count.**
After dismissal the next request will re-count and the prompt updates,
but within the same turn the model still sees `openAlertsCount = N` from
the request start, so it may keep harping about the dismissed alert.
Cosmetic, but explains "why does it still mention the alert I just
dismissed".

**H7. `resolveClient` URL-lock warning logic is inverted.**
`tools/types.ts:91-98`: it warns when the requested name's first word
ISN'T in the locked client name. But locked-client behavior already
returns the locked client regardless. The warning fires constantly when
the model passes the right name but with extra words ("Dr Calvin's
Clinic" first word "Dr" is not in "Calvin's Clinic"). False-positive
log noise.

### MEDIUM

**M1. `companion_messages` is still being written every turn**
(`index.ts:825-841`) alongside the new `assistant_messages` dual-write.
Fine for now, but the 50-msg pruning window is per-client and unrelated
to the thread model — when an agency owner has 200 turns across 5
threads they all share the same client_id and old turns get evicted out
of order.

**M2. `find_viral_videos` admin-only scout hint is leaked anyway.**
`index.ts:1001-1023` gates by `isAdmin` but the ENTIRE block is added
to the tool_result content. Models love to relay verbatim. The
"do NOT share with non-admin users" guard inside the string is
client-side trust at best.

**M3. `bulk_*` tools have no concurrency control or batched writes.**
`bulk_schedule_posts` / `bulk_reschedule_posts` /
`bulk_update_lead_status` all loop with `await` per item. 14 items =
14 round-trips. Not catastrophic, but combined with the 5-round
multi-call loop and Anthropic latency, a "schedule next 7 days for 3
clients" plan can easily push a single user request past 30s.

**M4. `generate_week_plan` and `draft_lead_outreach` fire Haiku calls
inside the main multi-round loop (each round budget is already 4096
tokens of Sonnet output).** If the model decides to call
`generate_week_plan` in round 1, then `bulk_schedule_posts` in round 2,
then explain in round 3 — that's Sonnet+Haiku+Sonnet+Sonnet on one user
turn. No per-call budget; no warning. A confused model in Auto mode
loop can burn $0.50+/message.

**M5. `get_open_alerts` "OR" filter syntax may not use indexes
optimally.** `intelligence.ts:203` uses
`q.or('user_id.eq.X,client_id.in.(...)')`. PostgREST OR clauses across
columns rarely use the same index — the `companion_alerts_open`
partial index is on `(user_id, severity, created_at desc)`. The
`client_id.in()` half does a separate scan. Fine at small scale; will
hurt as the table grows.

**M6. `parseScriptDraft` (CommandCenter.tsx:49-62) false-positives on
educational replies.** If the model explains the framework
("HOOK: a punchy first line. BODY: 3-4 beats. CTA: …") in plain text
the regex fires and creates an empty preview card. Tighten by requiring
the strict `TITLE:\nHOOK:` ordering produced by `draft_script` and
nothing else.

**M7. Build-trigger heuristic
(`build-mode.ts:723-727`) routes 3-word messages into build mode even
when the user clearly switched topics.** "show my pipeline" is 3 words →
goes to build mode if a session exists. The trigger should require BOTH
short message AND existing-build keywords like "yes/save/keep/skip".

**M8. `companion_alerts` auto-dismiss for `revenue_behind` never
fires.** The `monthly_progress > 0.7` gate only INSERTS in the second
half of the month, but no clean-up clause exists for revenue alerts.
Fix at the SQL level: dismiss when a new month rolls over.

**M9. `useActiveChat`'s `wasUpdatedRecently` only fires once on mount
(`CompanionDrawer.tsx:89-94`).** A user who hits Back inside the SPA
while a stale `connecta_active_chat` is still <60s old will always get
the drawer auto-popped. Mostly fine, but during demos this looks
twitchy. Consider also clearing `connecta_active_chat` on explicit
"new chat".

**M10. Build mode's `clearBuildProgress` is never called.**
`build-tool-handlers.ts:73-83` — only declared. Progress messages
accumulate in the thread. Probably intentional (the user sees what the
AI did), but they double up with the final reply.

### LOW

**L1. `get_finances` returns a Markdown table** (`finances.ts:111-122`),
violating system-prompt rule 1 ("no markdown"). The tool result is
context for the model, but the model often pastes it back verbatim.

**L2. System prompt rule 14 forbids "TOFU/MOFU/BOFU" but the prompt
header itself uses TOFU/MOFU/BOFU under "WHAT CONNECTA DOES".**
(index.ts:765-766 vs 795). Contradiction. Either drop them from the
methodology block or add "for your info, never repeat to the user".

**L3. `add_video_to_canvas` (build-tool-handlers.ts:568-640) writes
the new node at fixed `x=200, y=200` plus a y-stride** — over time
canvases pile up nodes off-screen. Inline `add_video_to_canvas` in
index.ts uses `x: 50, y: rowY`. Not the same coordinates → two AI
surfaces lay out canvas content differently.

**L4. `update_client_strategy` validation accepts mix percentages
that sum to 99 silently.** Probably fine, but worth a one-line note in
the tool result so the dashboard's "100%" pie doesn't visually drift.

**L5. `dualWriteCompanionTurn` can race with build-mode's own message
inserts** because build-mode writes user+assistant directly and
companion-chat falls through to dual-write at the bottom on the
non-build path. If both fire (shouldn't, but the route check is
heuristic), you get duplicate user messages.

**L6. `subscriber_clients`-based access uses `is_primary=true` only**
when picking the fallback client (index.ts:522-538). A subscriber with
multiple assignments and no primary lands on the owner-fallback path,
which then 400s if they own nothing.

**L7. `find_viral_videos` admin scout hints depend on `isAdmin`, but
the surrounding TOOL is exposed to every user.** Non-admins see the
"hashtags to try" string less the URL — but the model may still leak
the URL pattern from prior context. Flag `find_viral_videos` itself
admin-only if the hint matters.

### FYI

- `get_open_alerts` returns alerts with raw UUIDs in the body
  (`intelligence.ts:216-218`). Fine for the model, ugly if it pastes
  them back.
- `build-mode`'s `BUILD_TRIGGER` regex matches "I need a script idea"
  but also "let's create some scripts later". Not currently a bug, but
  the trigger surface is wider than the description suggests.
- `analytics.ts:233, 384` early-returns "No clients to scan" when
  `userClients.length === 0`. For an admin with no owned clients but
  managing the agency, this is wrong — should fall through to
  `accessibleClientIds`.

## What's working well

- **Multi-round loop in `index.ts:856-1609`** is genuinely solid: error
  handling on Anthropic 4xx/5xx, partial-reply append, fallback on
  unhandled tools, forced-text final round if MAX_ROUNDS hit. Best
  piece of code in the file.
- **`resolveClient` (`tools/types.ts:84-174`)** — clean access model,
  multi-strategy fuzzy match, well-commented. Same approach is
  duplicated in `handleResolveClient` (build-mode) — that's
  intentional and correct, both check admin + accessible IDs.
- **`save_script` validation + rollback in
  `build-tool-handlers.ts:902-1005`** — exactly what the ghost-script
  bug needed. Validate, dedupe, insert script, insert lines, rollback
  parent on lines failure. Bonus: structured log.
- **`companion_alerts` schema with partial unique index for dedupe and
  per-kind auto-dismiss** is a thoughtful design. Will scale.
- **`useActiveChat` cross-tab + intra-tab sync via storage event +
  CustomEvent** — exactly the pattern this needs.
- **`build-mode.ts` URL pre-processing + parallel transcription with
  `Promise.race` 40s timeout** is robust. Nice fallback to caption
  when transcription fails.
- **System prompt's `${urlClientId ? "ACTIVE CLIENT" :
  "AGENCY VIEW"}` block** at index.ts:773-775 — clear instructions,
  agency-mode never confuses the agency owner with their default
  client.

## What I'd defer

- Plan-proposal frontend card (B1 fix is more important than visual
  polish; the synthetic-message approach works minimally now).
- Bulk-tool batch SQL (M3) — current per-row loops are slow but
  correct; batch UPSERT only matters if users are doing 10+ writes
  routinely.
- M5 alert OR-filter index optimization — table is tiny.
- L3 canvas coordinate harmonization — cosmetic.
- Refactoring the inline `submit_to_editing_queue` to forward to
  build-mode (H4) — keep both, just harmonize naming if you don't
  want to deprecate.

Two unrelated cleanups that came up while reading: `companion_messages`
is now redundant with `assistant_messages` (dual-write should become
single-write soon — index.ts:825-841 + 1648-1660); the
`finance_transactions.user_id` scoping is correct, but
`get_revenue_vs_goal`'s per-client-strategy loop should be a single
join.
