```
                                     │   └ dual-write + actions[]    │
                                     └───────────────────────────────┘
```

Every chat surface (Bubble drawer, `/ai` CommandCenter, canvas threads) hits one POST to `companion-chat`. The fn resolves the caller's access set (admin vs subscriber via `subscriber_clients`), routes to a focused `build-mode` flow on build triggers or active sessions, otherwise runs a 5-round tool-use loop against `claude-sonnet-4-6` with ~65 tools. UI state flows back via `assistant_messages` Realtime. `useActiveChat` is the localStorage glue keeping the same thread under the user across pages.

## 3. The 5 things actually wrong right now (BLOCKER/HIGH only)

1. **BLOCKER — `delete_memory`/`list_memories` double-registered against the wrong table.** `tools/client.ts:20-39, 65-99` ships them writing `companion_state.workflow_context`; `tools/memories.ts:31-52` ships the correct `assistant_memories` versions. `index.ts:1583` runs `handleClientTool` BEFORE `handleMemoryTool`, so the buggy duplicates shadow the new ones. Every `list_memories` reads the dead JSONB scratch; every `delete_memory` silently no-ops on the real store. The assistant lies to itself about what it remembers.

2. **HIGH — Legacy `companion_messages` dual-written every turn but never read.** `index.ts:827-843, 1658-1662` insert + prune to 50 rows; history loads from `assistant_messages` by thread_id (`index.ts:666-668`). Dead write amplification on every request; 101 zombie rows.

3. **HIGH — `pending_plans` has 0 rows in 5 days.** Preview-Big-Actions is shipped end-to-end (table, tools, inline card UI) and unused. Rule 18c is rule 19 of 19, competing with `autonomy_mode === "auto"` forcing `tool_choice: any` at `index.ts:871` — auto mode actively kills the preview flow.

4. **HIGH — `open_client` and inline `get_client_info` (`index.ts:1516-1567`) ignore the agency access model.** Both still scope by `eq("user_id", user.id)`, not `accessibleClientIds`. Admins navigating to a subscriber's client by name hit "No client found"; subscribers can't reach their primary client this way. Pattern was fixed in tool modules, missed inline.

5. **HIGH — Build-mode's `preProcessUrls` doesn't pass `isAdmin` (`build-mode.ts:231-234`).** Multi-URL framework drops while admin-impersonating a subscriber likely fail in `handleAddUrlToViralDatabase`'s dedupe. Visible: drop 3 reels, get "0 added."

## 4. The 5 highest-leverage NEXT moves (ranked)

1. **Kill duplicate memory tools + drop `companion_messages` dual-write.** Delete the dupes from `tools/client.ts`, change handler order so memory module wins regardless, drop the `companion_messages` insert/prune. ~30 min. Unblocks section 3 #1 + #2 — stops the assistant from lying to itself about memory.

2. **Merge `CompanionDrawer` + `CommandCenter` chat plumbing into one hook.** Both reimplement thread load, realtime subscribe, optimistic send, action dispatcher. Lift into `useAssistantConversation(threadId)`. ~2-3 hrs. Any future feature lands in both surfaces; `plan_proposal` handling stops drifting.

3. **Rewrite the system prompt around two axes, not 19 rules.** Today: 19 rules + strategy + memory + brand blocks. Move (a) tenancy/identity into tool descriptions, (b) autonomy contract into one paragraph keyed on `autonomy_mode`, (c) stylistic "never say utilize" rules into a tail. ~30% token cut AND a quality bump from removing competing rules. **#3 because mid-tier output is the loudest pain in the vision quote.**

4. **Fix the autonomy/plan-proposal incoherence.** Either flip default off "auto" so rule 18c can fire, OR make 18c run in auto mode for destructive subset (`delete_script`, `send_contract`, `mark_post_published`, bulk ops ≥5). You shipped preview UI for zero users. ~1 hr wire + an evening to validate. Unlocks the user's stated "preview big actions" default.

5. **Multi-client weekly planning end-to-end.** `generate_week_plan` + `bulk_schedule_posts` exist; no Sunday trigger, no batch-across-N-clients UI. Add an `/ai` "Plan this week" preset that fires `generate_week_plan` per active client and renders one combined `propose_plan` card. ~4-6 hrs. **#5 because this IS workflow #2 from the vision and the plumbing is 80% there.**

## 6. Honest answer on memory cost-optimization

**No.** Premature.

10 memory rows across all users — the 40-row LRU cap isn't close to firing. `loadMemoriesForPrompt` is 200-500 tokens per turn worst-case. The real cost driver is the 19-rule prompt + strategy block + brand block + 40 history messages. And section 3.1 says it loud: your memory subsystem isn't expensive, it's *wrong* — `list_memories` returns dead-table data. Optimizing a system that's lying to itself is the wrong order.

Order: kill the duplicate (#4.1) → fix the autonomy/plan disconnect (#4.4) → prompt restructure (#4.3). After that, the highest-leverage cost win isn't trimming memory rows — it's `cache_control: {type: "ephemeral"}` on the static prefix (system + tools + brand/strategy). With 60+ commits/week of iteration, a prompt cache pays for itself in a day and is a 5-10× bigger win than memory trimming would ever be.
